# LLMOps (PoC)

Практики вокруг **жизненного цикла LLM** в системе: модели, промпты, наблюдаемость, стоимость, безопасность и связь с offline/online eval. Релевантно для агентского трека: решение по риску остаётся детерминированным; LLM — объяснение при фиксированных фактах.

---

## 1. Модели и версии

| Назначение | Модель (текущий PoC) | Где задаётся |
|------------|----------------------|--------------|
| Чат / JSON-объяснение | `gpt-4o-mini` (или переопределение `OPENAI_LLM_MODEL`) | `lib/config.ts`, env |
| Эмбеддинги политики | `text-embedding-3-small` | `lib/config.ts` |
| Валидация ответа API | Zod-схема на стороне приложения | `lib/llm/explain.ts`, `lib/types.ts` |

**Правила:** имя модели и версия политики (`policy_version`) фиксируются в логах/метаданных отчёта; при смене embedding-модели — переиндексировать чанки (в PoC: сброс in-memory кэша при деплое).

---

## 2. Промпты

- Системный и пользовательский промпт для объяснения — **код как источник истины**: `lib/llm/explain.ts`.
- Текстовый шаблон при деградации LLM — `templateExplanation` в том же модуле.
- Человекочитаемая выжимка и пример заполнения — `docs/prompts.md`.

**Изменения:** правки промпта через PR; для production — хранение версии промпта (hash или semver) в логах вызова LLM.

---

## 3. Вызовы API и формат ответа

- Используется **Chat Completions** с `response_format: { type: "json_object" }`; ожидаемая структура — см. `docs/prompts.md` и Zod `ResponseSchema` в коде.
- Парсинг JSON + `safeParse`; при ошибке — fallback без повторной «догадки» модели (шаблон + флаг `llm_degraded`).
- Один **retry** с укороченным контекстом политики (top-1 чанк) — см. `generateExplanationWithRetry`.

---

## 4. Наблюдаемость и стоимость

| Сигнал | Что логировать (без сырых ПДн) |
|--------|--------------------------------|
| Стоимость | `model`, `tokens_in`, `tokens_out` (из ответа API), агрегаты по дню |
| Качество шага | `llm_degraded`, причина (parse error / exception) |
| Связь с отчётом | `decision`, `flags`, `policy_version` |

В PoC логирование в stdout опционально; в production — единый формат (JSON) и запрет полного текста промпта с PII.

---

## 5. Безопасность и соответствие

- **Pre-guardrail** до вызова дорогих шагов: `lib/guardrails/pre.ts`.
- **Post-guardrail** на текст: `lib/guardrails/post.ts`.
- Решение `decision` в ответе пользователю **не** берётся из свободного текста LLM — только из rules engine + инварианты.

---

## 6. Eval и регрессии

- Offline-набор и пороги — `docs/specs/observability-evals.md` (golden dataset, метрики).
- Регрессия: прогон фиксированных запросов после смены промпта или модели; сравнение `decision` с эталоном rules engine и чеклистом по полям отчёта.
