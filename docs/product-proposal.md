# Product Proposal: AI Credit Risk Analyst

## 1. Обоснование идеи

Рынок кредитования сталкивается с ростом объёма заявок при ограниченном числе квалифицированных риск-аналитиков. Ручной анализ требует:

- Сбора данных из нескольких источников (БКИ, скоринг, внутренние регламенты)
- Интерпретации скоринговых показателей и сопоставления с политикой банка
- Формулировки обоснованного заключения для каждого заёмщика

Это приводит к задержкам, субъективности и высокой стоимости обработки. AI-агент позволяет:

- **Ускорить первичный анализ** — данные собираются и агрегируются автоматически
- **Стандартизировать оценку** — единые критерии и шаблоны снижают разброс в решениях
- **Объяснять риск** — LLM формирует понятное обоснование для аналитика и при необходимости для клиента
- **Снизить операционные риски** — меньше ручного ввода, меньше упущений при проверке политики

## 2. Цели и метрики

### Продуктовые

| Метрика | Целевое значение | Описание |
|---------|-------------------|----------|
| Время первичного анализа | < 2 мин на заявку | От запроса до готового risk report |
| Доля заявок без эскалации | 40–60% | Заявки, по которым агент даёт достаточную рекомендацию |
| Удовлетворённость аналитиков | ≥ 30 NPS | По результатам опроса пользователей |

### Агентские

| Метрика | Целевое значение | Описание |
|---------|-------------------|----------|
| Точность policy check | ≥ 95% | Корректность проверки по кредитной политике |
| Полнота объяснений | ≥ 90% | Наличие обоснования по ключевым факторам риска |
| Корректность решений | ≥ 85% | Совпадение рекомендации с экспертным решением на тестовой выборке |

### Технические

| Метрика | Целевое значение | Описание |
|---------|-------------------|----------|

| Стоимость на заявку | < $0.50 | LLM + инфраструктура |

## 3. Потенциальные сценарии использования

### 3.1 Основные сценарии

1. **Оценка заявки по запросу**  
   Аналитик пишет: «Оцени клиента 12345 для кредита 500k». Агент получает данные, вызывает PD-модель, проверяет policy, формирует risk report с объяснением и рекомендацией.

2. **Уточнение причин отказа/одобрения**  
   Аналитик спрашивает: «Почему высокий риск?» — агент объясняет на основе контекста заявки и политики.

3. **Проверка соответствия политике**  
   Аналитик запрашивает: «Проверь клиента по policy» — агент использует RAG для поиска релевантных правил и формирует чеклист.

### 3.2 Edge cases

| Сценарий | Поведение |
|----------|-----------|
| **Нет кредитной истории** | Агент помечает в отчёте, использует альтернативные данные (если есть), рекомендует ручную проверку или доп. документы |
| **Конфликт данных** | Фиксирует противоречия, снижает уверенность, рекомендует эскалацию |
| **PD-модель не отвечает** | Retry, при неудаче — fallback на rule-based оценку, флаг в отчёте |
| **Лимит превышен** | Автоматический reject с обоснованием по политике |
| **Неполные данные заявки** | Указывает недостающие поля, рекомендация — запросить доп. документы |
| **Запрос вне контекста** | Вежливый отказ, предложение переформулировать |

### Технические

|-------------|---------------------|

## 5. Архитектурный набросок

### 5.1 Модули и интеграции

```
┌─────────────────────────────────────────────────────────────────┐
│                     Credit Risk Analyst Agent                    │
├─────────────────────────────────────────────────────────────────┤
│  Инструменты:                                                     │
│  • get_credit_history  — API кредитной истории (БКИ)              │
│  • call_pd_model       — Внутренняя PD-модель (Probability of Default) │
│  • check_policy       — RAG с кредитной политикой банка          │
│  • generate_risk_report — Формирование risk report                │
└─────────────────────────────────────────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
┌───────────────┐    ┌───────────────┐    ┌───────────────┐
│ API кредитной │    │ Внутренняя    │    │ RAG: кредитная │
│ истории       │    │ база скорингов│    │ политика банка │
│ (БКИ)         │    │ (PD-модель)   │    │                │
└───────────────┘    └───────────────┘    └───────────────┘
        │                     │                     │
        └─────────────────────┼─────────────────────┘
                              ▼
                    ┌───────────────┐
                    │ LLM: объяснение│
                    │ риска, отчёт   │
                    └───────────────┘
```

### 5.2 Интеграции

| Компонент | Назначение |
|-----------|------------|
| API кредитной истории | Получение данных о кредитах, просрочках, долговой нагрузке |
| Внутренняя база скорингов (PD-модель) | Расчёт вероятности дефолта, risk grade |
| RAG с кредитной политикой | Поиск релевантных правил, проверка соответствия |
| LLM (OpenAI / Anthropic / локальная) | Генерация объяснений, формулировка рекомендаций |

## 6. Потенциальный data flow: что делегировать LLM/Agent, что нет

### Можно делегировать агенту / LLM

| Задача | Обоснование |
|--------|-------------|
| Запрос кредитной истории | Агент вызывает API по client_id |
| Вызов PD-модели | Агент передаёт данные, получает скоринговый балл |
| Проверка по policy | RAG + агент интерпретирует правила и сопоставляет с данными |
| Формулировка объяснения риска | LLM генерирует понятный текст на основе структурированных данных |
| Формирование risk report | Агент агрегирует данные и формирует итоговый отчёт |
| Ответы на уточняющие вопросы | Диалог в естественном языке |

### Нельзя делегировать (детерминированная логика)

| Задача | Обоснование |
|--------|-------------|
| Расчёт PD по формуле | Воспроизводимость, аудит, регуляторика |
| Финальное решение по выдаче кредита | Юридическая ответственность — только человек |
| Валидация и санитизация ввода | Безопасность, защита от инъекций |
| Аутентификация и авторизация | Критичные для безопасности операции |
| Запись в production-БД | Транзакции, контроль доступа |
